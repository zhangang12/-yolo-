## 2. 系统架构（关键认知）

整个系统是 **"四个大脑"协作**，不是一个模型包打天下：

```
拖入PDF → ①识别 ────────→ ②结构化 → ③规范比对 → ④原图标注
          (眼睛)            (算尺子)    (judge)      (手)
          YOLO + OCR        几何+图算法  规则引擎      坐标回写
```

| 模块 | 干什么 | 技术 |
|---|---|---|
| ①识别 | 找出元素、读文字 | **YOLO-seg**(面:防火分区/区域) + **YOLO检测**(对象:出口/楼梯/扶梯/闸机/商铺/门) + **OCR**(数字文字) |
| ②结构化 | 像素→实际尺寸，空间归属 | 几何计算 + **比例尺标定** |
| ③规范比对 | 撞阈值、判违规、分类 | **规则引擎**（确定性 if-else，可追溯，**不用 AI**） |
| ④标注 | 问题位置画回 PDF | 坐标反映射 + PyMuPDF 注释 |
| 疏散路径 | 任一点→出口最短路 | **图算法**(A*/Dijkstra)，非 ML |

**核心结论（务必传达给接手人）：**
- **YOLO 只占"识别"一环**。判违规靠规则引擎，算距离靠图算法，读数字靠 OCR。
- **大模型（VLM）不进核心链路**——它坐标会飘、会幻觉、不可复现，恰是审图最忌讳的。最多在"读文字增强""写报告文字"当辅助。MVP 阶段可零大模型。

## 5. 关键技术决策（踩过的坑，接手人勿重蹈）

1. **5 类任务 ≠ 一个 YOLO**。区域类(防火分区/公共区)用 **YOLO-seg 分割**(要算面积，必须闭合多边形)；对象类(出口/楼梯/闸机)用 **YOLO 检测框**；数字用 **OCR**；疏散路径用**图算法**。一个 YOLO-seg 模型可同时输出框和多边形，不必训两个。

2. **宽度 = 矢量几何实测值**，不是人工填、也不是纯 OCR。线长 × 比例尺 = 净宽。**门净宽 = 洞口宽 − 150mm**（规范口径，见 `docs/rule_engine_notes.md`）。

3. **尺寸线保持 polyline，不要转成矩形框**——线转框会变成"零面积"，YOLO 训不了。数字才用框(`val_text`)，线只用于定位/几何量测。

4. **比例尺要按"视口/区域"标定**，不能全图一个数：主平面图 1:150、楼梯详图可能 1:50，混用会让详图尺寸全错。

5. **规范比对用规则引擎，不用 AI**：消防规范是确定性数字阈值，要可解释、可追溯、不能编。

6. **优先要"矢量源文件"（PDF/CAD/DWG）**：矢量坐标精确，墙线/尺寸线/区域可直接抽，能让标注从"白纸上画"变成"审核机器预标"。标注单位之前用的是栅格 JPEG，发挥不出这个优势。

## 10. YOLO 训练流程速查

```
定类别 → 标注(切片1024²) → 转YOLO格式 → 划分8:1:1 → 训练 → 评估 → 找错例补标 → 重训(循环)
```
- 用 `ultralytics`，从 `yolo11s-seg.pt` 预训练权重迁移学习，`imgsz=1024`（目标小，必须大图+切片）。
- 每类 ≥150–300 实例起步；**一致性 > 数量**。
- 关键是"模型找错→人工补难例→重训"的主动学习循环。
- 类别表见 `docs/label_schema.md`。
