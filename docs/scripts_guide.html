1.3 系统级依赖(超出 requirements.txt)

Python 包(requirements.txt)以外,以下脚本依赖系统级程序:

系统程序谁用不装的后果
Tesseract OCR tools/e2e_demo.py 第①阶段 / app/backend/prelabel_pro.py --ocr 报 ❌ 未检测到 tesseract(OCR 引擎);矢量 PDF 文字层直读不受影响
CUDA + 驱动 app/backend/train_yolo.py --device 0 / tools/sahi_predict.py --device 0 torch.cuda.is_available()==False,自动降级到 CPU(慢 8-15×)

Tesseract 安装(三平台)

# Windows: 装 UB-Mannheim/tesseract 安装包,把 C:\Program Files\Tesseract-OCR 加入 PATH
#          中文图纸:安装时勾选 "Additional script data → Chinese Simplified"
#          地址: https://github.com/UB-Mannheim/tesseract/wiki

# macOS:
brew install tesseract tesseract-lang

# Ubuntu / Debian:
sudo apt install tesseract-ocr tesseract-ocr-chi-sim

# 验证: 任意终端
tesseract --version
本项目优先走"矢量文字层直读"(vector_extract.py)— 矢量 PDF 不依赖 tesseract。 只有无文字层的栅格 PDF 或 prelabel_pro --ocr 才真正需要 OCR。 不装 tesseract,矢量 PDF 工作流完全跑通。

tesseract_init.py — PATH 没设也能用 新

真实情况:不少 Windows 用户装了 Tesseract 但忘了加 PATH, tesseract --version 报"不是内部或外部命令"。 本项目在 tools/tesseract_init.py 加了自动探测:

from tesseract_init import ensure_tesseract
ok, cmd = ensure_tesseract(verbose=True)
# ✅ tesseract 在 C:\Program Files\Tesseract-OCR\tesseract.exe (tesseract v5.5.0.20241111)

探测优先级:shutil.which("tesseract")(PATH)→ Windows / macOS / Linux 常见安装目录。 所有用到 OCR 的脚本(e2e_demo.py、prelabel_pro.py)顶部都接了它。 只要 Tesseract 装在默认目录(Win 是 C:\Program Files\Tesseract-OCR\、 Mac 是 /opt/homebrew/bin/、Linux 是 /usr/bin/), 无需手工配 PATH 项目就能跑。

独立验证:

python tools/tesseract_init.py
# 输出实际找到的 tesseract.exe 路径 + 版本号

3. 几何与结构化(2026-06 新工具)

3.1 scale_calibrate.py — 比例尺自动标定

原理:打开 PDF 第一页 → 矢量文字层 grep 1:N → 用 PDF 物理 pt 大小 + jpg 像素大小 → 算 m/px。

# 单独运行(出 m/pt;给 --jpg-size 才出 m/px)
python tools/scale_calibrate.py  图纸.pdf  --jpg-size 9395 3339

典型输出:

{
  "source": "pdf_vector_text",
  "pdf_size_pt": [3382.0, 1202.0],
  "primary_ratio": 200,
  "ratio_freq": {"200": 2, "800": 1},
  "rendered_dpi": 200,
  "m_per_px": 0.025398
}

常见比例 → m/px(@ 200 DPI 渲染):

PDF 比例用途m/px @ 200DPI
1:150详细站厅(少见)0.0191
1:200站厅层平面图(主)0.0254
1:500总平面图(主)0.0635
1:1000区域示意0.1270

3.2 geom_measure.py — 几何量测

# 自动标定(标注有 dimension_val + polyline)
python tools/geom_measure.py  标注.xml  --out measure.json

# 手动 scale
python tools/geom_measure.py  标注.xml  --scale 0.0254  --out measure.json

核心 API(可 import):

函数用途
polygon_area_m2(pts, scale)Shoelace 多边形面积 × scale²
polyline_length_m(pts, scale)折线累计长度 × scale
door_net_width_m(pts, scale)洞口宽 − 0.15 = 门净宽
corridor_clear_width_m(pts, scale)通道/楼梯净宽 = 线长 × scale
associate_width_lines(shapes)width_line 关联最近的 fire_door / stair / gate

3.4 naming.py — 产物文件名标准化 2026-06

从图纸文件名(常含设计院图号、全角斜杠等噪声)抽出"站名 + 图类型",作为审查产物的简洁命名前缀。 mvp_e2e 和客户端 page_e2e 共用,保证产物命名一致。

python tools/naming.py    # 自测 9 个真实样本

# Python API
from naming import product_stem
product_stem("5╱17╱D01╱S╱S02╱COO╱QT╱010003╱A_车站-嘉宾站站厅层平面示意图.pdf")
# → "嘉宾站-站厅层"

三层抽取链:

  1. _KNOWN_STATIONS(已知站名 token 表,精度 100%):明海/珠光/贝尔路/黄君山/上油松/石龙/侨社/嘉宾/欧洲城
  2. 中文正则兜底:2-5 字"X站"(排除"车站"、"号X站"误匹配)
  3. 图类型:总平面图/站厅层/站台层/平面图/平面布置图/全局平面图

扩展新站:只改 tools/naming.py 第 14 行的 _KNOWN_STATIONS 列表(追加),mvp_e2e 和客户端立刻生效。

3.3 anno_to_structured.py — MVP 中枢适配器

这是规则引擎能跑真实数据的关键。把 CVAT 源标注 → rules/schema.md 定义的全部消费对象:

  • fire_compartment ← polygon + zone_type + area_m2_design
  • door ← fire_door + class + swing_dir + 关联 width_line 算净宽
  • exit ← safety_exit + 派生 zone / direction_group / leads_to
  • exit_pair ← 出口两两 same_direction / adjacent 判定
  • corridor ← stair_escalator / gate + width_line
  • evac_distance_line ← 折线 text_content 或几何长度
  • shop / shop_pair / fire_shutter ← 对应源对象
  • station ← 项目级配置(transfer_lines / project_all_doors_class_A 等)
python tools/anno_to_structured.py  标注.xml  --scale 0.0254  \
    --station-meta examples/station_meta.json  --out structured.json

5. 端到端

5.1 mvp_e2e.py — 一条命令 推荐

支持两种数据源:① CVAT 真值标注;② YOLO 模型自动识别(经 yolo_to_cvat 转成临时 CVAT 后走全套流程)。

# 方式①:CVAT 真值标注
python tools/mvp_e2e.py  annotations.xml  底图.pdf  输出目录  --pdf 底图.pdf

# 方式②:YOLO 自动识别(第一个位置参数留空字符串)
python tools/mvp_e2e.py  ""  底图.pdf  输出目录  --pdf 底图.pdf  \
    --yolo-weights best.pt  --yolo-conf 0.25  --yolo-device 0

# 手动 scale + 站点级配置(可选)
python tools/mvp_e2e.py  annotations.xml  底图.pdf  输出目录  \
    --scale 0.0254  --station-meta examples/station_meta.json

新参数:

参数说明
--yolo-weightsbest.pt 路径。提供后,xml 留空字符串即可走 YOLO 路径
--yolo-confYOLO 置信度阈值(默认 0.25)
--yolo-iouNMS IoU 阈值(默认 0.5)
--yolo-tile / --yolo-overlap切片大小/重叠(默认 1024 / 128)
--yolo-deviceGPU 设备号(默认 0;CPU 用 "cpu")

产物(命名前缀由 naming.product_stem 算,如 嘉宾站-站厅层_xxx):

文件说明
<stem>_e2e_fail.jpg底图 + 红框 FAIL + 右上角"全局违规"徽章
<stem>_structured.json12 类规则消费对象
<stem>_findings.json规则评估完整结果
<stem>_report.mdMarkdown 简报
<stem>_审查意见表.docx ⭐仿真实审查报告格式的可交付 Word(自动调 mvp_report_docx 生成)
<stem>_yolo_auto.xml (仅方式②)YOLO 推理转出的临时 CVAT

6. YOLO 训练

6.1 build_dataset.py — CVAT → YOLO-seg 数据集

python app/backend/build_dataset.py  标注目录  数据集目录  \
    --tile 1024  --overlap 128  --seed 0

标注目录结构要求:

标注目录/
  *.xml          (CVAT 1.1)
  images/        或 images/default/   (CVAT 1.1 嵌套)
    *.jpg

训练目标类(CLASS_NAMES,16 类,顺序即类别 ID):

fire_compartment, commercial_shop, surrounding_building,
station_exit_ground, vent_group_ground,
fire_door, stair_escalator, draft_curtain,
building_meta, vent_meta, dimension_val,
room_title, val_text,
safety_exit, gate, fire_shutter

polyline 类(fire_clearance_line / evac_distance_line / width_dimension_line)不作为训练目标(无面积无法 YOLO-seg),自动跳过。

6.2 train_yolo.py — 训练 / 评估

# 训练
python app/backend/train_yolo.py  train  data.yaml  \
    --model yolo11s-seg.pt  --epochs 50  --imgsz 1024  --batch 4  \
    --patience 10  --device 0  --workers 0  --no-amp  \
    --project runs  --name fire_seg_v1

# 评估
python app/backend/train_yolo.py  val  data.yaml  \
    --weights runs/fire_seg_v1/weights/best.pt  --imgsz 1024
RTX 50 系新卡 + Windows + 中文路径 三件套必加: --workers 0(避共享内存 bug)+ --no-amp(避 AMP cudaErrorUnknown)+ --batch 4(关 AMP 后显存翻倍)。

6.3 sahi_predict.py — 切片推理 新

解决:大图(14000×3300)直接缩到 1024 后小目标(防火门、闸机)大量漏检。

python tools/sahi_predict.py  best.pt  图.pdf  \
    --tile 1024  --overlap 128  --conf 0.25 --iou 0.5  \
    --out 输出目录

实测对比:嘉宾站站厅层(9395×3339)— 整图缩 1024 推理 = 1 个对象; 切片推理(44 块) = 114 个对象。数量级飞跃。

输入支持 PDF:检测到 .pdf 后缀自动 fitz 200DPI 渲染,中文路径安全。

6.4 yolo_to_cvat.py — YOLO 预测转 CVAT XML 2026-06

定位:把 best.pt 真正接入 mvp_e2e 端到端流程的适配器。 内部调 sahi_predict 切片推理,再把 detection list 转成 CVAT 1.1 XML(label / box / polygon 与 anno_to_structured 完全一致),下游 mvp_e2e 当 CVAT 真值喂入。

python tools/yolo_to_cvat.py  best.pt  底图.pdf  输出.xml  \
    --conf 0.25 --iou 0.5 --tile 1024 --overlap 128 --device 0

YOLO 识别不到的必填属性(LABEL_DEFAULT_ATTRS)自动赋默认值:

  • fire_door.class = unknown / swing_dir = 顺着疏散方向
  • fire_compartment.zone_type = 公共区
  • surrounding_building.building_type = unknown / fire_rating = unknown
  • vent_group_ground.discharge_type = unknown

规则引擎对这些 unknown 走 review_required(待人工复核)而不是硬判 FAIL — 这是正确行为。

客户端调用:mvp_e2e.py "" 底图.pdf out --yolo-weights best.pt 内部就走这条路径。

7. 报告导出

7.1 mvp_report_docx.py — 仿真实审查报告格式 推荐

仿照真实"消防设计文件审查意见表"(参考 J2025-004 系列)格式,产出可交付的 Word 文档:

python tools/mvp_report_docx.py  findings.json  e2e_fail.jpg  out.docx  \
    --station-name "嘉宾站(站厅层)"  --design-stage "初审 (AI 预审)"  \
    --source-pdf-name "嘉宾站站厅层平面示意图.pdf"

docx 内含 6 块:

  1. 标题"消防设计图纸 AI 预审意见表" + 副标题(自动生成,供审查参考,不替代法定审查结论)
  2. 基本信息表 4×4(工程名/审查阶段/图纸文件/审查日期/审查人系统/规则引擎版本/数据来源/审查范围)
  3. 审查结论(总检查项/通过/不合规/待人工复核,带颜色,粗体结论)
  4. 不合规明细表 6 列:序号 | 图上编号 | 中文审查意见 | 规范出处 | 设计回复(空) | 复核情况(□通过 □不通过)
  5. 嵌入标注图(对应"图上编号"列)
  6. 说明与免责

中文规则映射:30+ 条规则的英文 rule_id 映射到中文人话审查意见模板,如:

WIDTH-DOOR-EVAC-001 → "{tid} 疏散门净宽 {value}m,不满足规范要求 ≥{threshold}m,应加大门洞宽度。"

mvp_e2e 末尾自动调用本工具产出 <stem>_审查意见表.docx,无需单独跑。

7.2 report_docx.py — 旧版报告(兼容)

python tools/report_docx.py  e2e_输出目录

旧版 e2e_demo 产物对应的 Word 报告生成器。新流程已被 mvp_report_docx 取代,仅保留向后兼容。

8. 矢量层直读 vector_extract.py

直读图纸文字层里的"防火分区 名称+区域类型+面积",产出规则引擎格式数据。 比 OCR 准、不依赖标注/模型。

python tools/vector_extract.py  站厅层.pdf  --rules rules/rules.json

设计院常把"防火分区一(公共区):面积 3861.71㎡"直接写在矢量 PDF 文字层。 本工具直读这些文字 — 但这是设计院声称值,会标注来源并做自洽检查; 真伪几何复核留待后续。
