2. 增量训练策略

2.1 两种主流策略对比

策略命令时间适用
A. 从已有 best.pt 继续训
推荐增量场景		 --model best.pt 短(20-40% 全量时间) 新数据 < 旧数据 50% / 类别不变 / 想快速验证
B. 从预训练权重全量重训 --model yolo11s-seg.pt 长(完整 50-100 epoch) 新数据 >= 旧数据 50% / 加新类别 / 改 CLASS_NAMES / 定期"刷新"
推荐节奏:每收到 2-3 个新站标注就用 A 增量训一次(快速反馈); 每攒到 8-10 个新站后做一次 B 全量重训(避免增量训漂移)。

2.2 策略 A 工作机制

已有 best.pt (37 epoch,18 张图训出) ↓ 加载权重 (Transferred 555/561 items from pretrained weights) ↓ 新+旧合并的全量数据集 (重新切片) ↓ 再训 30-50 epoch ↓ 产出 best_v2.pt (mAP 期望 ↑)
易踩坑:策略 A 必须把新数据和旧数据合并到一个目录再 build_dataset。 单跑新数据会导致模型"灾难性遗忘"旧站点。

3. 增量数据准备

3.1 标准目录结构(推荐)

把所有批次的标注合并放到一个根目录:

data/anno_all/
├── images/                       ← 所有底图放一起
│   ├── 嘉宾站站厅.jpg               (老批次)
│   ├── 嘉宾站总平面.jpg
│   ├── ...
│   └── 新A站站厅.jpg                (新批次)
├── batch_20260610_hall.xml      ← 老批次 CVAT XML
├── batch_20260610_site.xml
└── batch_20260715_new_hall.xml  ← 新批次

关键:build_dataset.py 用 glob(*.xml) 扫描所有 XML, 自动遍历每个 <image> 并去 images/ 找对应底图。 所以 XML 文件可以多个共存,底图全在一个 images/ 下。

3.2 把新批次合并进来(具体步骤)

① 解压 CVAT 导出的 zip

# CVAT 1.1 标准结构: images/default/*.jpg + annotations.xml
unzip 新批次_站厅层.zip -d ./temp_new/

② 把图复制到 anno_all/images/

# Python(中文路径安全)
python -c "
import os, shutil
src = './temp_new/images/default'
dst = './data/anno_all/images'
os.makedirs(dst, exist_ok=True)
for f in os.listdir(src):
    s = os.path.join(src, f)
    if os.path.isfile(s):
        shutil.copy(s, os.path.join(dst, f))
"

③ 把 XML 重命名后复制

cp ./temp_new/annotations.xml ./data/anno_all/batch_$(date +%Y%m%d)_hall.xml

④ 跑质检确认新批次干净

# 单独跑新批次的 QC
python tools/fire_anno_tool.py qc ./data/anno_all/batch_20260715_new_hall.xml --type hall

ERROR 数必须 = 0 才能进训练集,否则会污染整批数据。

3.3 重建 YOLO 数据集

python app/backend/build_dataset.py  \
    ./data/anno_all/  \
    ./dataset/v2/  \
    --tile 1024 --overlap 128 --seed 0

关注产出报告里的"各类实例数":

  • 每类 ≥ 150(基线):mAP 应该可用
  • 30-149:可训但效果一般
  • < 30:大概率 mAP 接近 0,等于没训

4. 增量训练命令

4.1 策略 A:从 best.pt 继续训 默认

python app/backend/train_yolo.py  train  \
    ./dataset/v2/data.yaml  \
    --model models/fire_seg_gpu/best.pt  \
    --epochs 30 \
    --imgsz 1024 --batch 4 --patience 8  \
    --workers 0 --no-amp --device 0  \
    --project ./runs --name fire_seg_v2

关键点:

  • --model 指向已有 best.pt(不是 yolo11s-seg.pt)
  • --epochs 30:增量训不用 100 轮,30-50 够;--patience 8早停
  • --name fire_seg_v2:不要覆盖 v1,保留对比

4.2 策略 B:从预训练权重全量重训

python app/backend/train_yolo.py  train  \
    ./dataset/v2/data.yaml  \
    --model yolo11s-seg.pt  \
    --epochs 100 --imgsz 1024 --batch 4 --patience 20  \
    --workers 0 --no-amp --device 0  \
    --project ./runs --name fire_seg_v2_full

4.3 参数选择速查

硬件--model--imgsz--batch预估 50ep 时间
RTX 5060 / 8GByolo11s-seg.pt10244(关 AMP) / 8(开 AMP)20-40 min
RTX 4090 / 24GByolo11m-seg.pt10241610-15 min
RTX 3060 / 12GByolo11s-seg.pt1024830-60 min
CPU(任何)yolo11n-seg.pt64024-12 小时
RTX 50 系(Blackwell sm_120)必加
--no-amp(避 cudaErrorUnknown) + --workers 0(避 Windows 多进程共享内存崩溃) + --batch 4(关 AMP 后显存翻倍)。否则训到一半挂。

5. 评估与回归测试

5.1 跑 val

python app/backend/train_yolo.py  val  \
    ./dataset/v2/data.yaml  \
    --weights ./runs/fire_seg_v2/weights/best.pt  \
    --imgsz 1024

5.2 对照基线指标

把 v2 各类 mAP 和上面 §1.2 的基线表逐项对比:

  • ✅ 数据量大的类(fire_door / vent / gate)mAP 应 不降(±2% 内正常)
  • 🎯 之前 mAP=0 的类(safety_exit / commercial_shop)应 显著上升(到 0.3-0.7)
  • ⚠️ 全面下降:可能新数据有问题(质检漏检) → 单独 build 新批次跑一次 train 看哪类崩

5.3 视觉对比(强烈推荐)

同一张图,两个模型分别推理,人眼对比:

# 用 v1 模型
python tools/sahi_predict.py  models/fire_seg_gpu/best.pt  test_img.jpg  --out ./pred_v1/

# 用 v2 模型
python tools/sahi_predict.py  ./runs/fire_seg_v2/weights/best.pt  test_img.jpg  --out ./pred_v2/

# 打开两张 _pred.jpg 并排看

5.4 端到端回归测试

训完新模型后,把 mvp_e2e.py 跑一遍样例数据,确保下游链路不崩:

python tools/mvp_e2e.py  示例.xml  示例.jpg  /tmp/regression_v2/  --pdf 示例.pdf
# 看 _report.md 中违规清单数量、类型,与 v1 对比应该有变化但不混乱

6. 发布:用新权重替换旧的

6.1 新模型晋级入库(每版独立目录,不覆盖旧版)

约定:每个交付模型在仓库 models/ 下占一个目录,v1 永久保留在 models/fire_seg_gpu/, 新版另开 models/fire_seg_v2/,这样可随时回滚、可对照。

# 把训练出的新权重 + 评估图表晋级入库(只拷 .pt/.csv/.yaml/.png,绝不拷图纸 jpg)
mkdir -p models/fire_seg_v2/val
cp ./runs/fire_seg_v2/weights/best.pt  models/fire_seg_v2/best.pt
cp ./runs/fire_seg_v2/weights/last.pt  models/fire_seg_v2/last.pt
cp ./runs/fire_seg_v2/results.csv      models/fire_seg_v2/results.csv
cp ./runs/fire_seg_v2/args.yaml        models/fire_seg_v2/args.yaml
cp ./runs/fire_seg_v2_val/*.png        models/fire_seg_v2/val/   # 只有曲线/混淆矩阵是 png
绝不要把 train_batch*.jpg / val_batch*.jpg 拷进 models/ —— 它们是图纸拼图,属保密数据。.gitignore 已对 models/**/*.jpg 永久封禁作双保险。

6.2 切换生产权重

把客户端"方式② YOLO 权重"和 mvp_e2e --yolo-weights 指向新目录即可:

python tools/mvp_e2e.py "" 底图.pdf out/ --pdf 底图.pdf \
    --yolo-weights models/fire_seg_v2/best.pt

同步更新 DELIVERY.md §4.2 与 models/fire_seg_v2/README.md 的指标与路径。 回滚只需改回 models/fire_seg_gpu/best.pt。

6.3 客户端"YOLO 训练 → 评估"页测试

把 v2 best.pt 喂给客户端的评估页跑一遍,看混淆矩阵图。 若各类 mAP 持平或上升 → 发布;否则回滚到 v1。

7. 扩类别(改 CLASS_NAMES)的特殊注意

如果新批次包含原 CLASS_NAMES 没有的新类(如未来加 elevator), 处理路径:

7.1 改 build_dataset.py

# app/backend/build_dataset.py 第 30 行附近
CLASS_NAMES = [
    "fire_compartment", "commercial_shop", ...,  # 旧 16 个保持不变
    "safety_exit", "gate", "fire_shutter",
    "elevator",   # ← 追加到末尾,绝对不要插入中间
]
❌ 绝对不要做的:重排或删除 CLASS_NAMES 现有元素 — 这会让旧权重失效, 所有训过的类别 ID 全错位。新类只能追加到末尾。

7.2 同时改 fire_anno_tool.py 的 LABELS

给新标签加 schema 定义(geom / group / attrs / required), 否则质检会报 BAD_LABEL。同步改 rules/cvat_labels.json 让 CVAT 模板锁死下拉。

7.3 必须策略 B 全量重训

新类必须从 yolo11s-seg.pt 重头训(策略 A 的 best.pt 不认识新类 ID)。

8. 主动学习循环(进阶推荐)

YOLO 训练的"圣杯"流程 — 让模型自己暴露弱点,人工只补难例:

┌────────────────────────────────────────────────────────────┐ │ ① 用当前 best.pt 在【未标注的新站图】上 sahi_predict │ │ ↓ │ │ ② 看 conf 在 [0.25, 0.5] 区间的"低置信"预测 │ │ 这些是模型"拿不准"的样本,正是要补的难例 │ │ ↓ │ │ ③ 人工在 CVAT 复审、修正、确认(比从零标快 5×) │ │ ↓ │ │ ④ 加入训练集,策略 A 再训 30 epoch │ │ ↓ │ │ ⑤ 评估 → 弱点缩小 → 循环回 ① │ └────────────────────────────────────────────────────────────┘

每轮循环:

  • 新增 30-50 张图(2-3 个站)
  • 训练 30-40 min(策略 A)
  • mAP 期望 +2~5%(数据少的类提升最大)

9. 常见问题

Q1. 训到一半 RuntimeError: Couldn't open shared file mapping

Windows + 中文路径 + DataLoader 多 workers 触发。加 --workers 0 即可。

Q2. CUDA error: unknown error(RTX 50 系)

Blackwell sm_120 + AMP 偶发。加 --no-amp + --batch 4。

Q3. 训完 mAP 大幅下降

检查新批次质检报告:

  1. 标签是否在 schema 内(非法标签会被 build_dataset 跳过 → 训练丢实例)
  2. 每类实例数是否突变(看 build_dataset 报告)
  3. 底图分辨率是否一致(差距大需要调 tile 大小)
  4. val 集划分:固定 --seed 0 才能跨版本对比

Q4. 训练 OOM(显存不够)

降参数:--batch 2 --imgsz 768。或换更小模型 yolo11n-seg.pt。

Q5. 新加的类训完 mAP 还是 0

  1. 检查训练实例数:< 30 几乎不可能学会
  2. 检查 CLASS_NAMES 是否真的追加成功(看 data.yaml 里的 nc / names)
  3. 检查 build_dataset 报告里的"各类实例数"

Q6. 训练日志显示 "Transferred N/M items from pretrained weights"

正常 — 它表示从预训练权重迁移了 N 个层的权重,M-N 个层因为类别数变化需要重新初始化。 N/M 比例 > 90% 都正常。

Q7. 同一份数据 + 同样参数,两次跑 mAP 差别大

固定随机种子:--seed 0(build_dataset 也要传)。 val 集每次划分不同 = mAP 不可比。

附:增量训练 Cheatsheet

# 完整一轮增量训练(30 分钟)
# 假设新批次解压到 ./new_batch/,旧数据已在 ./data/anno_all/

# 1. 合并新批次到 anno_all
python -c "
import os, shutil
for f in os.listdir('./new_batch/images/default'):
    shutil.copy(os.path.join('./new_batch/images/default', f), './data/anno_all/images/')
shutil.copy('./new_batch/annotations.xml', './data/anno_all/batch_$(date +%Y%m%d).xml')
"

# 2. 质检(ERROR 必须 = 0)
python tools/fire_anno_tool.py qc ./data/anno_all/batch_$(date +%Y%m%d).xml

# 3. 重建数据集
python app/backend/build_dataset.py  \
    ./data/anno_all/  ./dataset/v2/  \
    --tile 1024 --overlap 128 --seed 0

# 4. 策略 A 增量训练
python app/backend/train_yolo.py train  \
    ./dataset/v2/data.yaml  \
    --model models/fire_seg_gpu/best.pt  \
    --epochs 30 --imgsz 1024 --batch 4 --patience 8  \
    --workers 0 --no-amp --device 0  \
    --project ./runs --name fire_seg_$(date +%Y%m%d)

# 5. 评估
python app/backend/train_yolo.py val  \
    ./dataset/v2/data.yaml  \
    --weights ./runs/fire_seg_$(date +%Y%m%d)/weights/best.pt

# 6. 与基线对比 → 决定是否发布
# 7. 发布:新模型晋级入库到独立目录 models/fire_seg_vN/(见 §6.1,勿覆盖 v1)
